# 仿真与实测对比及“实测崩”原因分析

## 一、数据文件与流程

| 文件 | 含义 |
|------|------|
| `PD-M1-v0_multi_joint_20260305-195346_exctra_sim.csv` | 激励轨迹 **仿真** 数据 |
| `PD-M1-v0_multi_joint_20260305-195432_exctra_real.csv` | 激励轨迹 **实测** 数据 |
| `PD-M1-v0_multi_joint_20260305-195641_sim_run.csv` | 跑步轨迹 **仿真** 数据 |
| `PD-M1-v0_multi_joint_20260305-195729_real_run.csv` | 跑步轨迹 **实测** 数据 |

**当前流程：**

1. 用 **激励轨迹仿真** 数据做最小参数辨识 → 得到 `X_hat`、`index_base`，保存于 `min_param_id_result.mat`。
2. 用 **跑步轨迹仿真** 数据做 FD/力矩验证 → 结果尚可。
3. 用 **跑步轨迹实测** 数据做 FD/力矩验证 → 出现“崩”（qdd 量级 10^8、τ 红黑相位与幅值差大）。

结论：**仿真数据没问题，问题集中在实测数据链。**

---

## 二、对比脚本用法

运行 `compare_sim_vs_real_trajectory.m`：

- **对比内容**：在同一条轨迹类型下，对 **仿真** 与 **实测** 做同一时间窗、同一预处理（窗 + qd 低通 + qdd 差分与平滑）后：
  - 绘制 **q、qd、qdd、τ** 的仿真 vs 实测曲线；
  - 绘制 **误差**（实测 − 仿真）随时间变化；
  - 打印各关节 **RMSE、max|误差|、MAE**。
- **模式**：脚本内 `compare_mode = 'running'` 时为跑步轨迹（sim_run vs real_run），改为 `'excitation'` 则为激励轨迹（exctra_sim vs exctra_real）。
- **时间窗**：`window_opts` 默认 `t_start_s=0, t_end_s=10`，与验证脚本一致，可按需修改。

通过对比可直接看到：同一时间段内，实测相对仿真的偏差在 q/qd/qdd/τ 各量上的大小与分布，便于定位是位置、速度、加速度还是力矩偏差更大。

## 三、导致“仿真 OK、实测崩”的主要原因

按工程上常见优先级归纳如下。

### 1. 辨识与验证数据源不一致

- 辨识用的是 **激励仿真**（理想动力学、无噪声、τ 与 q/qd 严格满足模型）。
- 验证用的是 **跑步实测**（真实传感器、电流估力矩、有噪声与延迟）。
- **X_hat** 拟合的是仿真世界的动力学；实机世界的 τ、q、qd 与仿真不同步、不同尺度时，**Y_min\*X_hat** 与 **τ_meas** 必然偏差大，FD 中 (τ_meas − h) 被放大，导致 **qdd_fd** 量级异常（如 10^8 rad/s²）。

### 2. 时间不同步

- 实机中 **q**（编码器）、**qd**（驱动器或差分）、**τ**（电流/力矩估计）往往来自不同硬件与采样时刻。
- 若 CSV 中“同一行”的 (q, qd, τ) 并非同一物理时刻，则 **Y(q,qd,qdd)\*X_hat** 与 **τ_meas** 在相位上对不齐，表现为红黑曲线相位差、峰谷错位。

### 3. qdd 由实测 q 二次差分得到，易放大噪声与单位错误

- 实测 q 有噪声，**qd = diff(q)/dt**、**qdd = diff(qd)/dt** 会大幅放大高频噪声。
- 若 **时间单位错误**（如 time 为 ms 却按 s 用）或 **dt** 异常（重复时间戳、采样不均），会导致 qdd 量级错误（例如本应 10～10² rad/s² 却出现 10^7～10^8）。
- 平滑可抑制噪声，但过度平滑会把真实加速度压掉，使“实际 qdd”偏小，而 FD 侧仍用 τ_meas 反推 qdd，两者不一致。

### 4. 力矩 τ_meas 的物理含义与模型不一致

- 实测 τ 多为 **电流换算** 或 **驱动器反馈**，存在：
  - **单位**（是否 N·m）、**符号**（某轴是否取反）、**偏置**（零点漂移）；
  - **滤波与限幅**（相位/幅值相对真实关节力矩有畸变）。
- 若 τ_meas 与模型约定不一致，会出现整体缩放或偏移，导致 (τ_meas − h) 系统性偏大，FD 解出的 qdd 爆炸。

### 5. 摩擦与未建模动力学

- 仿真多为理想刚体；实机有 **摩擦**（库仑、粘性）、**柔性**、**传动间隙** 等。
- 尤其在 **低速** 时，摩擦占主导，**Y_min\*X_hat** 只含惯性/重力/科氏，无法拟合摩擦项，τ 误差会明显。

---

## 四、建议排查顺序

1. **先核对 qdd 是否合理**  
   在生成 qdd 的代码中检查 **dt**、**time 单位**；画出 qdd 曲线，看量级是否在 10～10³ rad/s² 量级，若为 10^7～10^8 则必有问题。

2. **再查 τ 与 (q, qd) 的匹配关系**  
   确认 τ 单位、符号、时间戳是否与 q/qd 对齐；可用 **xcorr(τ_meas, τ_pred)** 看是否存在固定时移。

3. **做“重力段”验证**  
   选 qd≈0、qdd≈0 的片段，理论 τ ≈ g(q)。若此段 τ_meas 与 Y_min\*X_hat 仍差很多，优先怀疑 τ 单位/符号或重力项。

4. **利用本脚本的对比图与误差表**  
   看 **q/qd/qdd/τ** 中哪一项“实测−仿真”误差最大、是否集中在某几轴，再针对该项查数据来源与预处理。

## 五、脚本与报告对应关系

- **对比与绘图**：`compare_sim_vs_real_trajectory.m`（仿真 vs 实测 q/qd/qdd/τ 及误差）。
- **FD/力矩验证**：`run_full_dynamics_validation.m`（包含 FD/力矩对比）；需要单独调 π_fd 时用 `run_debug_pi_phys_id_fd_plot.m`。
- **辨识流程与验收**：见 `辨识流程与验证测试报告.md`。

将“仿真 vs 实测”的误差统计与曲线，与上述验证结果结合，即可系统判断问题来自 **数据（时间/单位/τ 定义）** 还是 **模型/辨识设定**。
